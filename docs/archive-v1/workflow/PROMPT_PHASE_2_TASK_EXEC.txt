## Workflow Phase 2 (6 bước)

### Bước 1: DISCOVER + DECIDE Tier 3 upload

Đọc TASKLIST row T0Y. Identify:
- Task scope: file gì emit, capability gì build
- Dependencies: T0(Y-1) DONE chưa, output ở đâu
- Pre-approved decisions áp dụng: D-NN nào relevant T0Y
- Conflicts cumulative: C-NN nào affect T0Y
- Stop conditions: ambiguity points cần human resolve trước khi code

Output DECIDE: "Cần upload Tier 3 thêm các files sau vì [rationale]:
                - [file 1] — [lý do]
                - [file 2] — [lý do]
                ...
                Total ~Y KB. OK?"

Đợi human confirm + upload.

### Bước 2: Emit Task Pack

Emit `taskpacks/<S-XX>-<T0Y>_<NAME>.md` per template:
- Read First: 3-8 file references (per Rule 3 evidence sources)
- Acceptance Criteria: bullet list cụ thể
- Sub-items breakdown: với effort per item
- Files emitted/modified: explicit paths
- Smoke test: command checks
- Stop Conditions: bullet list per Rule 5 STOP scenarios
- Pre-approved decisions referenced: D-NN + C-NN cumulative áp dụng

Đợi human ACK task pack scope trước khi proceed Bước 3.

### Bước 3: Code execution

AI emit code files theo Task Pack:
- Mỗi file create/modify emit vào sandbox
- Xuất ra /mnt/user-data/outputs/ cho human download
- Hoặc nếu code-heavy task, emit theo batches với commit logical

Track Stop Conditions throughout. Nếu hit STOP per Rule 5:
- Break code
- Write status
- Output: "STOP per Rule 5. Phát hiện [X]. Options:
   (a) Defer task này, làm task khác trước
   (b) Amend scope, human approve
   (c) Rollback, redo task pack
   Human decide."
- Đợi human decide → AI proceed. Decision logged Bước 6 amendment C-NN.

### Bước 4: Smoke test + verification

Chạy smoke commands theo Task Pack Acceptance Criteria. Capture output.
Document:
- All checks PASS: proceed Bước 5
- Some checks FAIL: document trong "Known issues" Bước 5 report

### Bước 5: Emit Implementation Report

Emit `reports/<S-XX>-<T0Y>_REPORT.md` per Rule 4 template:
- Scope: file gì emit, capability gì build
- Changed files: full path list
- Command output: build/test/lint logs
- Tests: smoke at minimum, document PASS/FAIL
- Known issues: tự khai báo, không giấu
- Bonus — Conflicts Surfaced: nếu phát hiện docs inconsistency mới Rule 7
- Verdict: 9 Gates PASS / FIX / REJECT preview

### Bước 6: DECIDE INDEX + decisions-log + TASKLIST updates — MANDATORY ATOMIC

CRITICAL RULE: Bước 6 emit cùng turn với Bước 5 report. KHÔNG split phiên.
File emit Bước 3+5 + INDEX patches Bước 6 phải là cùng atomic batch human save.

Output emission method (BUỘC tuân thủ):
- INDEX files đều >200 dòng, code files variable.
- KHÔNG dùng Cách 2 "tìm-thay đoạn" cho INDEX patches — human dễ copy sai → file
  corrupted (đã có bug pattern).
- BẮT BUỘC: AI đọc file gốc → apply edits trong sandbox → emit FULL REPLACE-READY
  FILE vào outputs/. Human download + save đè trực tiếp.

5 patches cần emit (đầy đủ file mới):

1. **INDEX_SLICES.md** — Full replace-ready:
   - Section [chữ cái] của <S-XX>: append per-task subsection mới
     "### Task T0Y — <Name>"
   - Subsection content:
     ```
     #### Artifacts (per-task T0Y)
     | Path | Type | What's inside |
     |---|---|---|
     | `taskpacks/<S-XX>-T0Y_<NAME>.md` | taskpack | <summary> |
     | `reports/<S-XX>-T0Y_REPORT.md` | implementation report | <summary> |
     | `<code file 1>` | code | <purpose> — see INDEX_PROJECT.md Section [letter] |
     | ... |

     **Summary:** 1 paragraph 3-6 câu cover scope + decisions applied + conflicts
     surfaced + verdict.
     ```
   - Update frontmatter: Total entries + Version bump if patch lớn
   - Append update log line nếu phiên này có patch lớn (vd C-NN amendment)

2. **INDEX_PROJECT.md** — Full replace-ready:
   - Add rows Section A/B/C/D theo path cho mỗi code file MỚI emit
   - Update "Last patched" column cho mỗi code file MODIFY
   - Update frontmatter: Total entries

3. **slices/<S-XX>_TASKLIST.md** — Full replace-ready:
   - T0Y row Status `TODO` → `DONE (YYYY-MM-DD)`
   - Append Status Log line cuối: "T0Y DONE Phiên N — verdict X"

4. **slices/<S-XX>_decisions-log.md** — Full replace-ready (nếu có mid-task C-NN):
   - Append entry vào Section 3 (Operational running log)
   - Full C-NN format: Surface date + Conflict + Source A/B + Resolution + Applied tasks + Status

5. **INDEX_DECISIONS.md** — Full replace-ready (nếu có mid-task C-NN ở patch 4):
   - Append row Section 2.2b (hoặc 2.3b nếu pattern S-00b dùng) cho C-NN mới
   - Update frontmatter Total entries

### Bước 7: Emit Review (9 Gates)

Emit `reviews/<S-XX>-<T0Y>_REVIEW.md`:
- 9 Gates verdict table với evidence per gate
- Overall verdict: PASS / FIX REQUIRED / REJECT
- Recommendations: nếu FIX REQUIRED, list revisions cụ thể

Nếu verdict FIX REQUIRED → AI propose fixes → human decide accept hoặc rollback.
Nếu PASS → task T0Y COMPLETE, proceed task tiếp theo.

### Bước 8: Verify (2-chiều)

Sau khi human save all files + 5 INDEX patches, AI request verify:

```bash
# 1. Task subsection trong INDEX_SLICES
grep -A1 "^### Task T0Y" INDEX_SLICES.md  # expect: header found

# 2. Code files reference INDEX_PROJECT
grep -oE '`<path-pattern>`' INDEX_PROJECT.md | sort -u

# 3. TASKLIST status
grep "T0Y" slices/<S-XX>_TASKLIST.md  # expect: DONE marker

# 4. Decisions-log + INDEX_DECISIONS sync nếu có mid-task C-NN
grep -c '^### [CD]-' slices/<S-XX>_decisions-log.md  # expect: N+1 if C-NN added

# 5. No corruption
grep -c "^### Task T" INDEX_SLICES.md  # expect: count increase by 1
```

Bất kỳ check fail → STOP, AI patch lại INDEX trước khi đóng task.

2. ĐỒNG THỜI append INDEX_DECISIONS.md Section 2.2b

Sanity check checklist cuối task T0Y:

Bất kỳ ✗ nào → outdated, patch trước khi đóng task.

MOCKUP AMBIGUITY RESOLUTION RULE (MAR-1)

Khi đọc mockup file (.html, .png, .fig, .sketch), nếu phát hiện

   - Tabs, segmented controls không có active state khác

   - Animation cues (drift, pulse, fade) — purely visual hay 
     có trigger state change?

2. Copy text mơ hồ:
   - Placeholder text trong input — cosmetic hay real seed data?

   - Success state — auto redirect hay manual click?

   - Demo hint card có xuất hiện ở production hay chỉ dev?

6. Responsive/breakpoint behavior:
   - Mockup mobile 414px — desktop layout thế nào?
   - Touch gesture (swipe, long-press) — có support không?

7. Accessibility cues:
   - Color-only differentiation — có cần icon/text fallback?
   - Focus states không hiện trong mockup — design thế nào?

## Phiên context — File system layout LAW (anh strict-enforce)

**Convention bắt buộc cho mọi turn của phiên này và chỉ áp dụng với tất cả các file .md:**

1. **`.md` docs (governance + spec + report + INDEX)** ALL ở `~/Downloads/` flat — KHÔNG subfolder. Path không carry semantic; **chỉ filename là canonical identifier**. Ví dụ:
   - `S-04_decisions-log.md`, `S-04_TASKLIST.md`, `INDEX_DECISIONS_S-04.md`, `03_API_CONTRACTS.md`, `LOG_CATALOG.md` ... ALL live ở `/home/hai-dang/Downloads/<filename>.md`
   - Đừng emit code with `slices/S-04_decisions-log.md` hay `docs/03_API_CONTRACTS.md` — đó là **suy luận sai về path**

2. **Code files** (apps/, packages/, infra/) → `~/projects/icpp/sicp/` THEO ĐÚNG cấu trúc thư mục — path subfolder CARRY semantic. Ví dụ:
   - `~/projects/icpp/sicp/apps/gateway/src/intent/intent.controller.ts`
   - `~/projects/icpp/sicp/packages/shared-types/src/sse/intent-stream.ts`
   - Đừng nhầm code file với .md doc

3. **Emit format quyết định theo file size + change footprint:**
   - **File ≤ 200 LOC** hoặc **≥ 30% lines change** → emit full file (download + copy/paste workflow)
   - **File ≥ 200 LOC** AND **< 30% lines change** AND **≥ 2 anchor points unique-verifiable** → emit Python **patch script** với defensive validation:
     - Đọc file → validate every anchor exists + is unique
     - Apply patches in-memory
     - Post-verify grep counts (min/max bounds)
     - Backup original to `.bak-<phiên>` extension
     - Atomic write (only if all checks PASS); exit code 1 + stderr nếu bất kỳ check nào fail
   - Quyết định emit format phải state explicit reasoning ở turn của Claude (vd: "File 1639 LOC, change ~12%, 7 anchor points all unique → emit patch script tiết kiệm ~40K tokens")

4. **Verify-before-edit (ER-R2 LAW áp dụng nghiêm)**: Mọi patch script bắt buộc:
   - Trước emit script: dùng `view`/`grep`/`sed -n` để đọc EXACT anchor strings hiện có trong file target
   - Tuyệt đối KHÔNG copy anchor string từ memory hay từ phiên trước — phải re-verify mỗi phiên
   - Script phải dry-run được trên file copy trước khi apply lên file thật

5. **Output emit cho patch script workflow:**
   - Tên file: `apply-<phiên>-batch-<N>.py` (vd `apply-batch-1.py`)
   - Mặc định DOCS_DIR hardcoded = `/home/hai-dang/Downloads` (KHÔNG path biến); user chỉ cần `python3 ~/Downloads/<script-name>.py` chạy từ bất kỳ pwd
   - Đầu script comment block ghi rõ Phiên + Files target + Number of sub-patches per file
   - Script tự emit summary cuối: "✅ Batch N atomic patch COMPLETE" hoặc "❌ FATAL: <reason>"
